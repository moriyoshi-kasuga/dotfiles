# 改善提案

**起点**: $ARGUMENTS

> ディレクトリパスが指定されている場合はそのディレクトリを探索の起点にします。
> テーマ（例: 「ディレクトリ構成」「型定義」「エラーハンドリング」）が指定されている場合はそのテーマに絞って調査します。
> 未指定の場合はカレントディレクトリ全体を俯瞰し、改善余地の大きい箇所を自律的に見つけます。

## 制約

- **実装・コード変更は行わない** — 提案・比較のみ
- **判断を押しつけない** — どの案を選ぶかはユーザーが決める
- 事実（コードの現状）と意見（改善の価値判断）を区別する

## 探索プロセス

1. **全体把握** — ディレクトリ構成・ファイル数・主要技術スタックを確認する
2. **テーマ特定** — 引数で指定されたテーマ、または探索から見えた課題領域を3〜5個に絞る
3. **詳細調査** — テーマごとに関連ファイルを読み込み、具体的な問題箇所・パターンを把握する
4. **提案の生成** — 各テーマに対して複数の改善案（最低2案）を策定し、比較軸を揃えて整理する

## 提案の比較軸

各提案は以下の軸で評価する：

| 軸 | 内容 |
|----|------|
| **難易度** | 低 / 中 / 高（実装コストの概算） |
| **影響範囲** | 局所（1ファイル以内）/ モジュール間 / システム全体 |
| **優先度** | 今すぐ / 次のイテレーション / 長期的に |
| **リスク** | デグレードや破壊的変更の可能性 |
