# 方針相談（planning）

**相談内容**: $ARGUMENTS

> - **困りごと・相談内容の指定**: それを中心の論点として調査する（例: 「エラーハンドリングが場当たり的で辛い」「この機能をどこに置くべきか迷っている」）
> - **対象パスの指定**: そのディレクトリ・ファイルを調査の起点にする
> - **両方指定**: 指定パスの範囲で、相談内容に絞って調査する
> - **未指定**: カレントディレクトリを俯瞰し、改善余地の大きい論点を自律的に見つけて同じ形式で提案する

## このスキルの位置づけ

| スキル | コード参照 | 出力 |
|---|---|---|
| `/think` | しない | 概念・設計パターンの議論 |
| **`/planning`** | **読み取りのみ** | **論点ごとの解決案の比較と推奨** |
| `/spec` | 読み取りのみ | 実装前提の仕様書 |
| `/rewrite` | 読み取り + 書き換え | ゼロベースの完全リライト |

`/planning` は「今あるものをどう直すか・どう進めるか迷っている」段階の相談窓口。方針が固まったら実装や `/spec` に引き継ぐ。

## 原則

- **実装・コード変更は行わない** — 提案と比較のみ。方針が決まった後の実行プラン作成や実装は行わない
- **困りごとから出発する** — 一般論の改善リストではなく、ユーザーが困っていることの解決を中心に据える。表面の症状ではなく根本原因を論点にする（例: 「コメントが多すぎる」の裏に「命名が意図を表現できていない」があればそちらを論点にする）
- **推奨案を必ず出す** — 選択肢を並べて判断を丸投げしない。自分が最も良いと考える案を A に置き、理由を添える。最終判断はユーザーに委ねる
- **事実と意見を区別する** — コードの現状は `ファイル名:行番号` で根拠を示し、価値判断（どうすべきか）と混ぜない
- **探索は広さ優先** — 全ファイルを読まず、論点に関係する代表ファイルをサンプリングして全体像を掴む

## 実行手順（調査と提案）

1. **相談内容の解釈** — 何に困っているのかを1〜2行で言語化し、独立した論点に分解する。解釈が複数ありえて調査方針が変わる場合のみ、調査前にユーザーへ確認する
2. **コード調査** — 論点ごとに関係するファイルを特定して読み、現状のパターン・問題箇所を `ファイル名:行番号` で押さえる
3. **選択肢の策定** — 各論点に対し解決案を2〜3案（A / B / C）作る。案を出す前に選択肢空間を広く洗い出し、そこから本当に検討に値するものだけに絞る
4. **提案の提示** — 「出力形式」に従って提示する。実行は終わり。

## 論点と選択肢の作り方

- **論点ごとに1番号** — 1つの番号には1つの独立した判断だけを入れる。番号間は互いに依存しないようにする
- **選択肢は A から順に** — 案が2つなら A / B、3つなら A / B / C。**A を必ず推奨案にする**
- **真に判断が分かれるものだけ論点にする** — どう考えても一択のものは論点にせず、「確定事項」として先に書く
- **利点・欠点は具体的に** — 「柔軟」「安全」のような抽象語で終わらせず、何が・どう変わるかを書く
- **欠点のない案を作らない** — 全案に欠点を明記する。欠点が書けない案があるなら、それは論点ではなく確定事項
- **推奨理由は差分で語る** — A の一般的な良さではなく、「B / C ではなく A である理由」を書く

## 出力形式

```
## 相談内容の理解
[ユーザーが何に困っていて、何が決まれば前に進めるのかを1〜3行で]

## 現状
[調査で把握した関連コードの現状を根拠付きで簡潔に。`ファイル名:行番号` を示す]

## 確定事項
[判断が一択で分かれない方針があればここに書く。なければ省略]

### 1. [論点のタイトル]
**問題**: [何が課題か。根拠となる箇所を添えて1〜3行]

- **A（推奨）**: [案の一言サマリ] — [何をどうするか]
  - 利点: [...] / 欠点: [...] / 工数: 小・中・大
- **B**: [案の一言サマリ] — [何をどうするか]
  - 利点: [...] / 欠点: [...] / 工数: 小・中・大
- **C**: [案の一言サマリ] — [何をどうするか]
  - 利点: [...] / 欠点: [...] / 工数: 小・中・大

**推奨理由**: [B / C ではなく A を推す理由。1〜2行]

### 2. [論点のタイトル]
...
```

提案後はユーザーの質問等に適宜答える。このスキルは方針の提案のみで、実行プランの作成や実装には携わらない。
