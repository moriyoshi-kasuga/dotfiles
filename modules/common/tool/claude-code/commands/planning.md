# 改善提案

**起点**: $ARGUMENTS

> - **ディレクトリパス指定**: そのディレクトリを探索の起点にする
> - **テーマ指定**（例: 「ディレクトリ構成」「型定義」「エラーハンドリング」）: テーマに絞って調査する
> - **両方指定**: 指定ディレクトリをテーマに沿って調査する
> - **未指定**: カレントディレクトリの構成・主要ファイルを俯瞰し、改善余地の大きい箇所を自律的に見つける

## 制約

- **実装・コード変更は行わない** — 提案・比較のみ
- **判断を押しつけない** — どの案を選ぶかはユーザーが決める
- **探索は深さより広さを優先** — 全ファイルを読み込まず、代表的なファイルをサンプリングして全体像を把握する
- 事実（コードの現状）と意見（改善の価値判断）を区別する

## 探索プロセス

1. **全体把握** — ディレクトリ構成・ファイル数・主要技術スタックを確認する（Glob/Bash で俯瞰、深読みは最小限）
2. **テーマ特定** — 引数で指定されたテーマ、または探索から見えた課題領域を3〜5個に絞る
3. **詳細調査** — テーマごとに代表ファイルを1〜3個読み込み、具体的な問題箇所・パターンを把握する
4. **提案の生成** — 各テーマに対して複数の改善案（最低2案）を策定し、比較軸を揃えて整理する

## 提案の比較軸

| 軸 | 内容 |
|----|------|
| **難易度** | 低 / 中 / 高（実装コストの概算） |
| **影響範囲** | 局所（1ファイル以内）/ モジュール間 / システム全体 |
| **優先度** | 今すぐ / 次のイテレーション / 長期的に |
| **リスク** | デグレードや破壊的変更の可能性 |
