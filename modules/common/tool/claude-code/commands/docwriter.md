# ドキュメント最小再構築（Rust特化）

**対象範囲**: $ARGUMENTS

## 前提

既存のコメント・ドキュメントはコードの変更に追従していないことが多く、当てにしない。
そのため既存の文章は参考にせず、現在のコードの型・シグネチャ・制御フローのみを根拠に、必要な部分だけをゼロから書き直します。
ドキュメントは書くほど保守コストになるため、量より正確さを優先し、原則として最小限に留めます。

## 方針

- 既存のコメントは一度すべてクリアしてから書き直す。既存の文章を要約・編集して引き継ぐことはしない。
- コードを読めば分かることは書かない。変数名・型・関数分割で表現できるなら、コメントは不要。
- 言語仕様や標準ライブラリの挙動、`Send`/`Sync` の一般的なイディオムなど、自明な内容の説明は書かない。
- 使い方や挙動を示したい場合は、長文の説明ではなく `# Examples` の doctest で表現する。
- 公開 API のみ doc コメント（`///`）を付ける。非公開の項目には基本的にコメントを付けない。
- `// SAFETY:` コメントは、呼び出し側・実装側が満たすべき非自明な前提条件がある場合のみ書く。
- どうしても文章が必要な場合も、既存の文章は読まず、コードから読み取れる最重要点のみを1行程度で書く。
- コードそのものは変更しない。対象はコメントと doctest のみ。

## 対象

- 行コメント（`//`）・ブロックコメント（`/* */`）
- doc コメント（`///` / `//!`）
- `unsafe` ブロックの `// SAFETY:` コメント

## 改行・整形規則

- 改行してよいのは、箇条書きの項目間、セクション見出しの前後、コードブロックの前後、および1文が150文字を超え途中で区切る必要がある場合のみ。
- 段落の途中に無駄な空行を挟まない。

## 実行手順

1. `$ARGUMENTS` から対象ファイル・箇所を確定する。未指定なら `git diff HEAD` の変更範囲を対象にする。
2. 対象範囲の既存コメント（`//`, `/* */`, `///`, `//!`）を一度すべて削除する。
3. コメントを削除したコード（型・シグネチャ・制御フロー）だけを見て、必要な doc コメント・SAFETY コメント・doctest をゼロから設計する。
4. 改行・整形規則に沿って文章を整える。
5. `cargo test --doc` を実行し、doctest が通ることを確認する。
6. 変更内容を報告する。

## 出力形式

### 変更内容のサマリー

- **削除した不要コメント**: どのファイルのどの箇所にあった、どういうコメントを削除したか
- **追加/改善した doctest**: ドキュメントの代わりに doctest で表現した仕様の要約
- **残した最小限のコメント/docコメント**: 残す必要があった WHY や SAFETY の説明

### 検証結果

`cargo test --doc` の実行結果と、不自然な改行がないか確認した旨を報告する。
