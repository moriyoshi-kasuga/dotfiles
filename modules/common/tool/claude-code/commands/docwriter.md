# ドキュメント最小化・再構築（Rust特化）

**対象範囲**: $ARGUMENTS

## 前提：コメントは負債である

コードは頻繁に変更されますが、コメントは自動的には追従しません。書かれた瞬間からコメントは嘘をつく（コードと乖離する）リスクを抱えます。
そのため、本スキルでは**「コメントは少ないほど良い」**という思想に基づき、以下の基準でドキュメントを再構築します。

1. **自己文書化されたコードが最良**: 適切な変数名、型設計、関数分割によって、コメントなしで意図が伝わるようにします。
2. **ボイラープレート・自明なイディオムにコメントは不要**:
   - 例: `UnsafeCell`をフィールドに持つ構造体に対して、安全性を検証した上で`Send`や`Sync`を`impl`する際、Rustの標準的なイディオムであるためコメントは一切不要です。
   - 他にも、言語仕様や標準ライブラリの挙動をなぞるだけのコメントはすべてノイズであり、削除対象です。
3. **詳細な仕様説明はdoctest（`# Examples`）に委ねる**:
   - 言葉による長文の説明は、コードの変更に伴って陳腐化しやすく、また誤解を招きやすいです。
   - コンパイラや `cargo test` によって検証される doctest (`# Examples`) に使い方やエッジケースの挙動を示し、docコメントのテキスト自体は要点の要約のみに絞り込みます。
4. **不自然な改行や過剰な修飾の徹底排除**:
   - AIが生成しがちな不自然な英語・日本語や、行末で `XXX. That\n` のように不適切にぶつ切りになった改行は一切許容しません。

## 方針（厳守）

- **過剰・不要なコメントの削除を最優先する**
  - コードを読めばわかる「WHAT（何をしているか）」や「HOW（どう実装しているか）」はすべて削除します。
  - 理由のわからない「WHY」が残っている場合も、コードの文脈（エラーハンドリングや型）から推測可能ならコメントは不要です。
- **ドキュメントの詳細はdoctestに逃がす**
  - 関数の使い方や挙動を示す場合、長文で説明するのではなく、`# Examples` に動作可能な doctest を書きます。
  - これにより、コード変更時にテストが落ちるため、ドキュメントの陳腐化を自動的に防ぐことができます。
- **不自然な改行・文章を完全に排除する**
  - 文の途中で意味なく改行されたり、英語の接続詞や助動詞の前後（例：`That\n`）でぶつ切りになったりする改行は絶対に避けてください。
  - 原則として、コメント内の1文は改行せずに1行で記述するか、日本語であれば読点（、）、英語であれば明確な節の区切り以外では改行しません。
- **コードそのものは一切変更しない** — 本スキルはコメントとdoctestのみを対象とします。

## 対象

- 行コメント（`//`）・ブロックコメント（`/* */`）
- doc コメント（`///` 各種宣言 / `//!` モジュール・クレート）
- `unsafe` ブロックの `// SAFETY:` コメント（真に非自明な安全性の前提条件がある場合のみ。自明なものは書かない）

## 書き直しの内容規則

### 通常コメント（`//`）
- **原則すべて削除**:
  - コードを読めば意図が伝わる部分は、コメントを完全に削除します。
  - 例外的に、ハードウェアの制約、外部ライブラリのバグ回避など、コードから絶対に読み取れない「非自明なWHY」がある場合のみ、1行〜数行の極めて短いコメントを残します。

### `unsafe` / `Send` / `Sync` などの実装コメント
- **ボイラープレートへのコメント禁止**:
  - `UnsafeCell` を使用していることによる `Send` / `Sync` の手動実装など、Rustコンパイラへの指示や標準的な型システム上のイディオムに対して「なぜこれが必要か」を説明するコメントは書かないでください。
- **SAFETYコメントの最小化**:
  - `// SAFETY:` コメントは、`unsafe` ブロックを使用する際に、呼び出し側や実装側が満たすべき「非自明な」安全対策・不変条件（例: 生ポインタのアライメント等）がある場合のみ記述します。
  - Rustのドキュメント慣習として必要とされる場合でも、冗長な説明は避け、要点のみを簡潔に書きます。

### doc コメント（`///` / `//!`）
- **公開 API のみに限定**: 非公開の構造体や関数には doc コメント (`///`) ではなく、極力コメントなし（必要な場合のみ最小限の `//`）とします。
- **テキスト説明の要約化**:
  - 概要は1行〜2行の短い文で表現します。
  - 詳細な仕様や引数の説明を長々とテキストで書くのではなく、`# Examples` に doctest を配置して示します。
- **doctestの重視**:
  - 複雑な制約やエッジケースは、`# Examples` 内の `assert_eq!` などを通じて表現します。
  - 既存の doctest がある場合も、長すぎる説明文は削り、テストコード自体を充実させます。
- **セクションの限定**:
  - パニック条件がある場合は `# Panics`、`unsafe fn` の場合は `# Safety` を記述します。ただし、これらも簡潔な箇条書きに留めます。

## 改行・整形規則（厳守）

AIが不自然な位置で改行することを防ぐため、以下の整形ルールを強制します。

- **1文＝1行を基本とする**:
  - コメントやdocコメント内の1文（句点「。」やピリオド「.」で終わる単位）は、原則として改行せず1行で書きます。
  - 画面幅に合わせるための機械的な折り返し（特に英語で `We use\n this function` や `It is safe because...\nThat` のようになるもの）は**絶対に禁止**します。
- **改行してよい位置**:
  - 箇条書き（`- `）の項目間
  - セクション見出し（`# Examples` など）の前後
  - コードブロック（` ``` `）の前後
  - 1文が極端に長く（150文字以上）、どうしても途中で区切る必要がある場合の、読点（、）や関係代名詞の直前など、文脈が完全に切り替わる位置
- **空白行の制限**:
  - 段落の途中に無駄な空行を挟まないでください。

## 実行手順

1. **対象範囲の確定** — `$ARGUMENTS` からファイル・箇所を特定する。未指定なら `git diff HEAD` の変更範囲を対象にする。
2. **現在のコードと既存コメントの分析**:
   - コード自体のシグネチャ、型、制御フローを読み解く。
   - 「このコメントは、コードを読めば本当に不要ではないか？」「型や関数名で表現できないか？」を自問する。
   - 既存のコメントが Rust の一般的なイディオム（`Send`/`Sync` impl や `UnsafeCell` 等）に対する説明であれば、**削除候補**とする。
3. **コメントの徹底削減と doctest 設計**:
   - 既存コメントの大半を削除する。
   - API の詳細な仕様は、テキストで説明する代わりに `# Examples` セクションを設計し、doctest のコード例で表現する。
4. **クリーンな文章作成と改行ルールの適用**:
   - 残す必要のある最小限の説明文は、1文＝1行で書き、不適切なぶつ切り改行がないか確認する。
5. **検証**:
   - `cargo test` を実行し、追加・修正した doctest が正常にパスすることを確認する。
   - `cargo clippy` と `cargo fmt` を実行してコードベースの品質を維持する。
6. **変更の報告**:
   - どの不要なコメントを削除したか、どのような doctest に置き換えたかを報告する。

## 出力形式

### 変更内容のサマリー

- **削除した不要コメント**: （どのファイルのどの行にあった、どういった不要コメント/イディオム解説を削除したか）
- **追加/改善した doctest**: （ドキュメントの代わりに doctest で表現した仕様の要約）
- **残した最小限のコメント/docコメント**: （どうしても残す必要があったWHYやSAFETYの説明）

### 検証結果

`cargo test` (doctest 含む) の実行結果と、不自然な改行がないかダブルチェックした旨の報告。
