# ドキュメントレビュー

**対象**: $ARGUMENTS

> - **ファイルパス指定**: そのファイルを読み込む
> - **URL指定**: WebFetch で取得する
> - **未指定**: 会話コンテキスト中のドキュメントを対象にする

## 制約

- **書き直しは行わない** — 明示的に求められない限りセクションを丸ごと書き直さない
- フィードバックは簡潔かつ実行可能な形で提供する
- 著者の既存の用語・表現を尊重し、正確に保持する
- 良い点も必ず言及する

## レビュー観点

ドキュメントの種類に応じて適用する観点を選ぶ。技術記事には全観点、仕様書・スキル定義には 3〜5 のみ適用するなど柔軟に判断する。

### 1. タイトル（技術記事向け）

- 言語・ライブラリ・技術名などの主要キーワードが含まれているか
- 内容を盛り込みすぎてタイトルがぼやけていないか

### 2. 結論ファースト（技術記事向け）

- 記事・各セクションの冒頭に結論・要点が来ているか（PREP法）
- ファーストビューで記事の要約が伝わるか

### 3. 流し読みのしやすさ

- 重要な箇所が **太字** で強調されているか（太字だけ拾い読みして大意が掴めるか）
- 複数項目が並ぶ箇所に箇条書きを使っているか
- 見出し階層を見るだけで構成が把握できるか
- 図・コードブロックが適切に使われているか

### 4. 前提の明示

- 主題（何が書いてあるか・書いていないか）が明確か
- 対象読者・前提知識のレベル感が伝わるか
- ヘッダーの不整合や用語の不一致がないか

### 5. トーン・構造

- 誰かを攻撃・批判するような表現になっていないか
- 構造上の矛盾・読みにくさ・情報の重複がないか

## 出力形式

### 良い点

[優れている点を列挙]

### 改善点

各指摘は以下の形式で記載する：

- **[観点] [重要度: 高/中/低]** 場所（任意）: 具体的な問題 → 改善提案

重要度の基準：

- **高**: 読者が意味を誤解する・必要な情報が欠落している
- **中**: 読みにくい・伝わりにくい・前提が不明確
- **低**: より良くなる表現・スタイルの提案

### サマリー

[全体評価と、優先して対応すべき点]
