# ドキュメントレビュー

技術記事・ドキュメントを以下の観点でレビューします。

## レビュー方針
- フィードバックは簡潔かつ実行可能な形で提供する
- 著者の既存の用語・表現を尊重し、正確に保持する
- 構造上の問題・矛盾・読みにくさに焦点を当てる
- 明示的に求められない限りセクションを書き直さない
- ヘッダーの不整合や用語の不一致を指摘する
- 対象読者を意識した表現になっているか確認する
- 良い点も必ず言及する

---

## レビュー観点

### 1. タイトル
- 言語・ライブラリ・技術名などの主要キーワードが含まれているか
- 興味を引くキャッチーな表現になっているか（煽りすぎは NG）
- 内容を盛り込みすぎて、タイトルがぼやけていないか

### 2. 結論ファースト
- 記事・各セクションの冒頭に結論・要点が来ているか（PREP法）
- ファーストビューで記事の要約が伝わるか
- 「詳細は以降で説明」という構成になっているか

### 3. 流し読みのしやすさ
- 重要な箇所が **太字** で強調されているか（太字だけ拾い読みして大意が掴めるか）
- 複数項目が並ぶ箇所に箇条書きを使っているか
- 目次（見出し階層）を見るだけで構成が把握できるか
- 図・コードブロックが適切に使われているか

### 4. 前提の明示
- 記事の主題（何が書いてあるか・書いていないか）が明確か
- **対象読者** が明示または自明か
- **前提知識** のレベル感が伝わるか
- 動作確認したバージョン・日付が記載されているか

### 5. トーン
- 誰かを攻撃・批判するような表現になっていないか
- 難しく堅すぎず、読者がポジティブな気持ちで読み進められるか

---

## 出力形式

### 良い点
[優れている点を列挙]

### 改善点
各指摘は以下の形式で記載する：
- **[観点] [重要度: 高/中/低]** 場所（任意）: 具体的な問題 → 改善提案

重要度の基準：
- **高**: 読者が意味を誤解する・必要な情報が欠落している
- **中**: 読みにくい・伝わりにくい・前提が不明確
- **低**: より良くなる表現・スタイルの提案

### サマリー
[全体評価と、優先して対応すべき点]
