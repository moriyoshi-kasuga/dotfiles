# リファクタリング

**対象**: $ARGUMENTS

> - **ファイルパス・箇所指定**: そのファイル・箇所を対象にする
> - **未指定**: 直前の作業対象または `git diff HEAD` の変更を対象にする

## 制約

- **機能変更は行わない** — 振る舞いを変える修正は提案のみとし、実施前にユーザーに確認する
- 一度に変更する範囲は最小限に留める
- 変更理由を明示する（なぜ改善か）

## 実行手順

1. **現状把握** — 対象コードを読んで構造・依存関係を把握する
2. **テスト確認** — テストが存在する場合は実行して green であることを確認する（存在しない場合はその旨を明記する）
3. **段階的変更** — 1つの観点ごとに小さく変更する
4. **テスト確認（変更後）** — テストが通ることを確認してから次の変更に進む

## チェックリスト

- 重複コードの統合（DRY原則）
- 長すぎる関数・メソッドの分割（単一責任）
- 変数名・関数名の意図が明確か
- マジックナンバーの定数化
- 不要なコメント・デッドコードの削除
- エラーハンドリングの一貫性
- 型・インターフェースの整理

## 出力形式

変更箇所ごとに：

- **変更理由**: なぜこの変更が改善か
- **変更前**: 問題のあるコード
- **変更後**: 改善されたコード

変更完了後: テスト結果（pass/fail）と変更の総括を示す。
