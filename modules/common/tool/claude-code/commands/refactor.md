# リファクタリング（Rust特化）

**対象**: $ARGUMENTS

> - **ファイルパス・箇所指定**: そのファイル・箇所を対象にする
> - **未指定**: 直前の作業対象または `git diff HEAD` の変更を対象にする

## このスキルの位置づけ

| スキル | 機能変更 | 既存設計の扱い | 主な用途 |
|---|---|---|---|
| **`refactor`** | **行わない** | **維持する** | 動いているコードの内部構造を安全に整理 |
| `brushup` | 行ってよい | 意図を最優先で尊重 | 荒削りな実装を全面的に磨き上げる |
| `rewrite` | 大胆に行う | 参考に留める | ゼロベースで最良の設計に完全に書き直す |

## 制約

- **機能変更は行わない** — 外から見た振る舞い（入出力・エラー・panic の有無・副作用・公開 API）を変える修正は実装せず、「リファクタリング範囲外の提案」で報告する
  - `unwrap()` → `?` への置き換えも「panic → エラー伝播」の挙動変更なので提案に回す
  - 公開 API（`pub` なシグネチャ・型・モジュール構成）の変更も提案に回す
- **気づきは握りつぶさない** — 作業中に見つけたバグ疑い・パフォーマンス問題・設計上の課題は、スコープ外でも必ず提案として残す
- 一度に変更する範囲は最小限に留め、変更理由（なぜ改善か）を明示する

## 実行手順

1. **現状把握** — 対象コードを読んで構造・依存関係を把握する
2. **グリーン確認** — `cargo check` と `cargo test`（対象クレート・モジュールに絞ってよい）で green を確認する
   - red のテストがある場合は、その箇所のリファクタリングを保留して報告する
   - テストが存在しない場合はその旨を明記し、変更前に最小限の特性テスト（characterization test）の追加を提案する。追加しないまま進める場合はリスクを明示する
3. **段階的変更** — 1つの観点ごとに小さく変更する
4. **変更後の確認** — 各変更後に `cargo check`、観点の区切りごとに `cargo test` と `cargo clippy` を通してから次へ進む
5. **仕上げ** — 最後に `cargo fmt` を実行し、`cargo clippy` の指摘が増えていないことを確認する
6. **気づきの記録** — 挙動変更を伴う改善点（バグ疑い・パフォーマンス・構造）はその場で実装せずメモし、提案セクションにまとめる

## チェックリスト（挙動を変えない範囲で）

### 構造

- 重複コードの統合（DRY原則）
- 長すぎる関数・メソッドの分割（単一責任）
- モジュール内の可視性の最小化（クレート内で閉じる `pub` → `pub(crate)` / 削除。公開 API に触れるものは提案へ）
- デッドコード・未使用 `use`・未使用変数の削除

### 命名（Rust慣習）

- snake_case（変数・関数・モジュール）/ CamelCase（型・トレイト）/ SCREAMING_SNAKE_CASE（定数）
- メソッド名の慣習: `is_*` / `has_*`（真偽値）、`as_*`（参照変換）、`to_*`（複製変換）、`into_*`（所有権移動）、`from_*` / `new` / `with_*`（構築）
- 略語・単体で意味不明な名前、同じ概念への複数名の混在がないか

### イディオム（等価な書き換え）

- `for` ループ → イテレータチェーン（`map` / `filter` / `fold` / `any` / `all`）
- `match` / `if let` / `let-else` / `matches!` の使い分け（1パターンのみ → `if let`、早期リターン → `let-else`、真偽値のみ → `matches!`）
- `contains_key` + `insert` → `entry().or_insert()` / `or_default()`
- `Option` / `Result` のコンビネータ（`map` / `and_then` / `unwrap_or_else` / `ok_or_else`）で等価に書けるか
- 参照で等価に書ける不要な `clone()` / `to_string()` の削除
- 型推論が効く箇所の冗長な型注釈の削除

### 型・トレイト

- 手書きのトレイト実装 → derive（`Debug` / `Clone` / `PartialEq` / `Default` など）
- 散在する変換ロジック → `From` / `TryFrom` 実装への集約
- マジックナンバー・マジック文字列 → `const` / `static`

### コメント方針

AI生成コードは過剰なコメントになりがちなので、リファクタリング時に積極的に整理する。

- **WHAT（何をしているか）は基本的に削除する** — 自明なコードへの説明コメントは冗長。複雑なアルゴリズムなど、コードを読むだけでは意図を追えない場合に限り残す
- **HOW（どう実装したか）はコメントでカバーしない** — 挙動の保証はコメントではなく doctest / テストで担保する。対応するテストがなければ、削除の前提として追加を検討する
- **WHY（なぜこの実装か）は変数名・関数名で表現する** — 命名に意図を込め、コメントに頼らない。名前で表現しきれない背景（仕様上の制約・既知の外部要因など）だけを残し、冗長な説明・同じ内容の繰り返しは書かない
- `unsafe` ブロックの `// SAFETY:` コメントは削除しない（不足していれば追加する）

## 出力形式

### 実施した変更

変更箇所ごとに：

- **変更観点**: 構造 / 命名 / イディオム / 型・トレイト / コメント
- **変更理由**: なぜこの変更が改善か
- **変更前**: 問題のあるコード（簡潔に）
- **変更後**: 改善されたコード（簡潔に）

### 検証結果

`cargo check` / `cargo test` / `cargo clippy` の結果と変更の総括。

### リファクタリング範囲外の提案

作業中に気づいた、振る舞い変更を伴うため今回は実施しなかった改善点。該当がなければ「なし」と明記する。

各提案は以下の形式で記載する：

- **[種別: バグ疑い/パフォーマンス/構造・設計]** `ファイル名:行番号` — 問題の説明 → 改善提案

種別の基準：

- **バグ疑い**: 既存実装の誤りと思われる挙動（エッジケースの考慮漏れ・`unwrap()` による panic 経路・エラーハンドリングの欠落など）。確証がない場合は「疑い」であることを明示する
- **パフォーマンス**: 不要なアロケーション・非効率なアルゴリズム・ホットパスの `clone()` など、振る舞いに影響しうる最適化
- **構造・設計**: モジュール分割・責務の再配置・公開 API の変更など、リファクタリングの範囲を超える構造改善

提案の実施可否はユーザーが判断する。実施を求められた場合のみ着手する。
