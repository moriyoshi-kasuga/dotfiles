# リファクタリング

**対象**: $ARGUMENTS

> - **ファイルパス・箇所指定**: そのファイル・箇所を対象にする
> - **未指定**: 直前の作業対象または `git diff HEAD` の変更を対象にする

## 制約

- **機能変更は行わない** — 振る舞いを変える修正は実装せず、出力の「リファクタリング範囲外の提案」で報告する
- **気づきは握りつぶさない** — 作業中に見つけたバグ疑い・パフォーマンス問題・設計上の課題は、スコープ外でも必ず提案として残す
- 一度に変更する範囲は最小限に留める
- 変更理由を明示する（なぜ改善か）

## 実行手順

1. **現状把握** — 対象コードを読んで構造・依存関係を把握する
2. **テスト確認** — テストが存在する場合は実行して green であることを確認する（red のテストがある場合は、その箇所のリファクタリングを保留して報告する）
   - テストが存在しない場合はその旨を明記し、変更前に最小限の特性テスト（characterization test）の追加を提案する。追加しないままリファクタリングする場合はリスクを明示する
3. **段階的変更** — 1つの観点ごとに小さく変更する
4. **テスト確認（変更後）** — テストが通ることを確認してから次の変更に進む
5. **気づきの記録** — 作業中に見つけた振る舞い変更を伴う改善点（バグ疑い・パフォーマンス・構造）は、その場で実装せずメモしておき、出力の提案セクションにまとめる

## チェックリスト

- 重複コードの統合（DRY原則）
- 長すぎる関数・メソッドの分割（単一責任）
- 変数名・関数名の意図が明確か
- マジックナンバーの定数化
- 不要なコメント・デッドコードの削除
- エラーハンドリングの一貫性
- 型・インターフェースの整理

## 出力形式

### 実施した変更

変更箇所ごとに：

- **変更理由**: なぜこの変更が改善か
- **変更前**: 問題のあるコード
- **変更後**: 改善されたコード

### テスト結果

[pass/fail と変更の総括]

### リファクタリング範囲外の提案

作業中に気づいた、振る舞い変更を伴うため今回は実施しなかった改善点。該当がなければ「なし」と明記する。

各提案は以下の形式で記載する：

- **[種別: バグ疑い/パフォーマンス/構造・設計]** `ファイル名:行番号` — 問題の説明 → 改善提案

種別の基準：

- **バグ疑い**: 既存実装の誤りと思われる挙動（エッジケースの考慮漏れ・エラーハンドリングの欠落など）。確証がない場合は「疑い」であることを明示する
- **パフォーマンス**: N+1・不要な計算・非効率なアルゴリズムなど、振る舞いに影響しうる最適化
- **構造・設計**: モジュール分割・責務の再配置・インターフェース変更など、リファクタリングの範囲を超える構造改善

提案の実施可否はユーザーが判断する。実施を求められた場合のみ着手する。
