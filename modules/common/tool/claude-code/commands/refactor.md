# リファクタリング

**対象**: $ARGUMENTS

> 引数が指定されている場合はそのファイル・箇所を対象にします。
> 未指定の場合は直前の作業対象または git diff (ステージ済み変更) を対象にします。

## 実行手順

1. 対象コードを読んで現状を把握する
2. テストが存在する場合は実行して green であることを確認する
3. 小さな変更を段階的に行う
4. 変更後にテストが通ることを確認する

## チェックリスト

- 重複コードの統合（DRY原則）
- 長すぎる関数・メソッドの分割（単一責任）
- 変数名・関数名の意図が明確か
- マジックナンバーの定数化
- 不要なコメント・デッドコードの削除
- エラーハンドリングの一貫性
- 型・インターフェースの整理

## 制約

- **機能変更は行わない** — 振る舞いを変えるリファクタリングは提案のみ
- 一度に変更する範囲は最小限に留める
- 変更理由を明示する（なぜ改善か）

## 出力形式

変更箇所ごとに：

- **変更前**: 問題のあるコード
- **変更後**: 改善されたコード
- **理由**: なぜこの変更が改善か
