# ブラッシュアップ

**対象**: $ARGUMENTS

> - **ファイルパス・箇所指定**: そのファイル・箇所を対象にする
> - **未指定**: 直前の作業対象または `git diff HEAD` の変更を対象にする

## このスキルの位置づけ

| スキル | 機能変更 | コード変更 | 主な用途 |
|---|---|---|---|
| `refactor` | 行わない | 行う | 動いているコードの内部構造を安全に整理 |
| `review-code` | 行わない | 行わない | 問題点の指摘・提案のみ |
| **`brushup`** | **行ってよい** | **行う** | 荒削りな実装を全面的に磨き上げる |

基盤ロジックは実装者の意図を尊重しつつ、細部の品質を全面的に引き上げる。

## 方針

- **実装者の意図を最優先にする** — アーキテクチャ・アルゴリズム・基本的な設計判断は変えない
- **動く → 良い** を目指す。すでに動いているコードをより読みやすく・堅牢に・保守しやすくする
- 機能的な改善（エラーハンドリングの追加・バリデーション強化など）も行ってよいが、**大きな仕様変更は提案として報告**する
- 変更は一度に行わず、観点ごとに段階的に適用する

## チェックリスト（優先順）

### 1. 命名
- 変数・関数・型の名前が意図を正確に伝えているか
- 略語・単体では意味不明な名前になっていないか
- 同じ概念に複数の名前が混在していないか

### 2. エラーハンドリング
- エラーが握りつぶされていないか（空の catch、無視された Result/Option）
- エラーメッセージに「何が・なぜ」が含まれているか（コンテキスト情報の付与）
- エラーの伝播方法が一貫しているか（panic vs Result vs 独自エラー型）
- リカバリ可能なエラーとそうでないものを区別しているか

### 3. コメント・ドキュメント
- 「何をするか」ではなく「なぜそうするか」が書かれているか
- 自明なコードへの冗長なコメントを削除する
- 公開 API・非自明な引数・副作用には説明を追加する
- TODO / FIXME が放置されている場合は対処または整理する

### 4. モジュール・構造分割
- 1つの関数・モジュールが複数の責務を持っていないか
- 100行を超える関数は分割できないか検討する
- 関連する処理がファイル・モジュールをまたいで散らばっていないか
- 再利用可能な処理が重複していないか

### 5. 型・インターフェース
- `any` / 過度に広い型を使っていないか
- 型で表現できる制約が文字列・コメントで表現されていないか
- 型エイリアス・構造体・列挙型で意図を明確にできないか

### 6. 細部の堅牢性
- 境界値・null / nil / 空コレクションの考慮漏れ
- マジックナンバー・マジック文字列の定数化
- デッドコード・未使用インポートの削除
- ログ出力のレベルと内容が適切か

## 実行手順

1. **現状把握** — 対象コードを読み、実装者の意図・基盤ロジックを把握する
2. **改善点の洗い出し** — チェックリストに沿って問題点を列挙する
3. **段階的ブラッシュアップ** — 観点ごとに変更を適用する
4. **大きな変更の判断** — アーキテクチャや仕様に影響する改善は実装せず提案として記録する

## 出力形式

### 実施した変更

変更箇所ごとに：

- **変更観点**: 命名 / エラーハンドリング / コメント / 構造分割 / 型 / 堅牢性
- **変更理由**: なぜこの変更が改善か
- **変更前**: 修正前のコード（簡潔に）
- **変更後**: 改善後のコード（簡潔に）

### 提案（未実施）

実装者の判断が必要な、仕様・設計に影響する改善案。該当がなければ「なし」と明記する。

各提案は以下の形式で記載する：

- **[種別: 機能追加/設計変更/破壊的変更]** `ファイル名:行番号` — 問題の説明 → 改善提案

提案の実施可否はユーザーが判断する。実施を求められた場合のみ着手する。
