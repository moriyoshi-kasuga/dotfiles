# ブラッシュアップ（Rust特化）

**対象**: $ARGUMENTS

> - **ファイルパス・箇所指定**: そのファイル・箇所を対象にする
> - **未指定**: 直前の作業対象または `git diff HEAD` の変更を対象にする

## このスキルの位置づけ

| スキル | 機能変更 | コード変更 | 主な用途 |
|---|---|---|---|
| `refactor` | 行わない | 行う | 動いているコードの内部構造を安全に整理 |
| `review-code` | 行わない | 行わない | 問題点の指摘・提案のみ |
| **`brushup`** | **行ってよい** | **行う** | 荒削りな実装を全面的に磨き上げる |

基盤ロジックは実装者の意図を尊重しつつ、細部の品質を全面的に引き上げる。

## 方針

- **実装者の意図を最優先にする** — アーキテクチャ・アルゴリズム・基本的な設計判断は変えない
- **動く → 良い** を目指す。すでに動いているコードをより読みやすく・堅牢に・保守しやすくする
- 機能的な改善（エラーハンドリングの追加・バリデーション強化など）も行ってよいが、**大きな仕様変更は提案として報告**する
- 変更は一度に行わず、観点ごとに段階的に適用する

## チェックリスト（優先順）

### 1. 命名（Rust慣習）

- snake_case（変数・関数・モジュール）/ CamelCase（型・トレイト）/ SCREAMING_SNAKE_CASE（定数）が守られているか
- メソッド名の慣習に沿っているか：
  - 真偽値 → `is_*` / `has_*` / `can_*`
  - 所有権を移動して変換 → `into_*`
  - 参照で変換（コスト低） → `as_*`
  - 値を複製して変換 → `to_*`
  - 構築 → `from_*` / `new` / `with_*`
- 型引数名: 単一なら `T`、複数なら意味のある名前（`K`, `V`, `E` など）
- 略語・単体では意味不明な名前になっていないか
- 同じ概念に複数の名前が混在していないか

### 2. エラーハンドリング

- `unwrap()` / `expect()` が本来 `?` で伝播すべき箇所で使われていないか
  - `expect()` を使う場合はメッセージに「なぜここで失敗しないはずか」を書く
  - テストコード・プロトタイプ以外での `unwrap()` は要検討
- `?` 演算子で伝播できる箇所を `match` / `unwrap` で書いていないか
- エラー型の設計：
  - ライブラリ → `thiserror` でカスタムエラー型
  - アプリケーション → `anyhow::Error` で文脈付きエラー（`.context()` / `.with_context()`）
- `map_err` / `ok_or` / `ok_or_else` を使い `Option` → `Result` の変換が適切か
- エラーのコンテキスト情報（何が・なぜ失敗したか）が含まれているか
- `Result` を返す関数で戻り値を無視していないか（`#[must_use]` の活用も検討）

### 3. 所有権・借用

- 不必要な `clone()` がないか
  - 参照で十分な箇所で所有権を取っていないか
  - `to_string()` の連発より `&str` / `Cow<str>` で済まないか
- 関数引数の型選択：
  - 読み取りのみ → `&T` / `&str` / `&[T]`（`&String` / `&Vec<T>` は不要に具体的）
  - 所有権が必要なとき → `T` / `String` / `Vec<T>`
  - 所有権か参照か呼び出し側に委ねる → `impl Into<T>` / `impl AsRef<T>`
- ライフタイム注釈が必要な箇所で省略されていないか（elision が効かない箇所）
- `Rc` / `Arc` の使用が適切か（マルチスレッドに `Rc` を使っていないか）

### 4. イディオム・イテレータ

- `for` ループをイテレータチェーンで書けないか（`map` / `filter` / `flat_map` / `fold` / `any` / `all`）
- インデックスアクセス（`v[i]`）を `iter().enumerate()` に置き換えられないか
- `match` / `if let` / `while let` / `matches!` の使い分けが適切か
  - 1パターンのみ → `if let`
  - 真偽値のみ → `matches!`
  - 複数パターン → `match`
- `collect::<Vec<_>>()` に型推論が効いているか（冗長な型注釈になっていないか）
- `Option` チェーンに `map` / `and_then` / `unwrap_or` / `unwrap_or_else` / `?` を使えるか
- `entry().or_insert()` / `entry().or_default()` パターンで `contains_key` + `insert` を置き換えられないか

### 5. 型・トレイト設計

- `Box<dyn Trait>` と `impl Trait`（静的ディスパッチ）の使い分けが適切か
- 有限なバリアントを `String` / `u8` で表現していないか → `enum` にできるか
- Newtype パターンで型安全性を高められないか（誤用を型で防ぐ）
- derive マクロで実装できるトレイトを手書きしていないか：
  - `Debug` / `Clone` / `Copy` / `PartialEq` / `Eq` / `Hash` / `Default` / `Display`
- `Default` トレイトを実装すべき型が実装していないか
- `From` / `Into` / `TryFrom` / `TryInto` で型変換を標準化できないか
- `Iterator` / `Display` / `FromStr` など標準トレイトを実装すると利便性が上がる箇所はないか

### 6. パフォーマンス・アロケーション

- ホットパスで不必要なヒープアロケーションがないか
  - `String::new()` + `push_str` vs. `format!` vs. `write!` の使い分け
  - `Vec::with_capacity()` で事前確保できるか
- `String` を返すより `impl Display` / `impl Write` にできないか
- `Box<[T]>` / `Rc<[T]>` など固定長が確定後に `Vec<T>` を使い続けていないか
- 引数に `&str` の代わりに `impl AsRef<str>` / `impl AsRef<Path>` を使えるか

### 7. コメント・ドキュメント

AI生成コードは過剰なコメントになりがちなので、ブラッシュアップ時に積極的に整理する。

- **WHAT（何をするか）は基本的に削除する** — 自明なコードへの説明コメントは冗長。複雑なアルゴリズムなど、コードを読むだけでは意図を追えない場合に限り残す
- **HOW（どう実装したか）はコメントでカバーしない** — 挙動の保証はコメントではなく doctest / テストで担保する。対応する doctest がなければ追加を検討する
- **WHY（なぜこの実装か）は変数名・関数名で表現する** — 命名に意図を込め、コメントに頼らない。名前で表現しきれない背景だけを残し、冗長な説明・同じ内容の繰り返しは書かない
- 公開 API のドキュメントコメントは目的と契約（引数・戻り値・エラー条件のうちシグネチャから読み取れないもの）に絞る
- `unsafe` ブロックに Safety コメント（`// SAFETY:`）があるか

### 8. 細部の堅牢性

- 境界値・空スライス・空 `Vec`・`None` の考慮漏れがないか
- マジックナンバー・マジック文字列を `const` / `static` に切り出せるか
- デッドコード・未使用 `use` / 未使用変数（`_` prefix か削除）
- `#[allow(...)]` に理由コメントがあるか
- `unsafe` の使用範囲が最小化されているか
- `panic!` / `unreachable!` / `todo!` / `unimplemented!` が本番コードに残っていないか

## 実行手順

1. **現状把握** — 対象コードを読み、実装者の意図・基盤ロジックを把握する
2. **改善点の洗い出し** — チェックリストに沿って問題点を列挙する
3. **段階的ブラッシュアップ** — 観点ごとに変更を適用する
4. **大きな変更の判断** — アーキテクチャや仕様に影響する改善は実装せず提案として記録する

## 出力形式

### 実施した変更

変更箇所ごとに：

- **変更観点**: 命名 / エラーハンドリング / 所有権・借用 / イディオム / 型・トレイト / パフォーマンス / コメント / 堅牢性
- **変更理由**: なぜこの変更が改善か
- **変更前**: 修正前のコード（簡潔に）
- **変更後**: 改善後のコード（簡潔に）

### 提案（未実施）

実装者の判断が必要な、仕様・設計に影響する改善案。該当がなければ「なし」と明記する。

各提案は以下の形式で記載する：

- **[種別: 機能追加/設計変更/破壊的変更]** `ファイル名:行番号` — 問題の説明 → 改善提案

提案の実施可否はユーザーが判断する。実施を求められた場合のみ着手する。
