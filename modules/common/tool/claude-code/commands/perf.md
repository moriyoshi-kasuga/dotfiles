# パフォーマンス調査

**対象**: $ARGUMENTS

> - **ファイルパス・処理指定**: その箇所のボトルネックを調査する
> - **症状の説明**（例: 「一覧取得が遅い」「メモリが増え続ける」）: 該当箇所を特定して調査する
> - **未指定**: 直前の会話コンテキストの対象を調査する

## 制約

- **コードの変更は行わない** — 調査と改善提案のみ
- 根拠なき推測を避ける — コードから確認できる事実に基づいて指摘する
- 「遅い」「重い」という定性評価だけでなく、計算量・クエリ回数・データ量など定量的な根拠を示す

## 調査プロセス

1. **症状の整理** — 遅い・重い・リークしているの具体的な状況を把握する
2. **コードの読み取り** — 対象箇所を読み取り、処理フロー・データフローを把握する
3. **ボトルネック候補の列挙** — 計算量・I/O・クエリ・メモリなど観点別に候補を挙げる
4. **証拠の確認** — ループ構造・クエリパターン・データ量の仮定から影響度を推定する
5. **改善案の策定** — 各ボトルネックに対して優先度付きで改善案を提示する

## 調査観点

- **計算量** — O(n²) 以上のループ・再帰・ネストした処理
- **N+1クエリ** — ループ内でのDB/API呼び出し
- **不要な処理** — キャッシュできる計算の繰り返し・不要なコピー
- **メモリ** — 大きなデータの保持・参照の解放漏れ・バッファリングの欠如
- **I/O** — 同期的なブロッキング処理・バッチ化できる個別リクエスト
- **インデックス** — DBクエリのフルスキャン・複合インデックスの欠如
