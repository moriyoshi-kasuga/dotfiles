# 外部調査

**対象**: $ARGUMENTS

> - **OSS名・技術名指定**: そのプロジェクトの設計・実装方針を調査する
> - **アーキテクチャのテーマ指定**（例: 「イベントソーシングの実装例」「認証基盤の設計比較」）: テーマに沿って複数のOSS・事例を横断調査する
> - **比較したい設計パターン指定**: 複数の選択肢を挙げて比較調査する
> - **未指定**: 直前の会話コンテキストから調査テーマを読み取る

## 制約

- **実装・コード変更は行わない** — 調査結果のまとめと参考情報の提示のみ
- **出典を必ず明示する** — 参照した記事・リポジトリ・ドキュメントのURLを示す
- 事実（一次情報から確認できること）と推測・意見（自身の解釈・評価）を明確に区別する
- 情報源の鮮度に注意し、古いバージョンの情報である可能性があれば明示する
- 単一の情報源に依存せず、複数の情報源を突き合わせて裏付けを取る

## 調査プロセス

1. **調査テーマの特定** — 引数・会話から調査対象の技術・OSS・比較軸を把握する
2. **一次情報の探索** — WebSearch で公式ドキュメント・GitHubリポジトリ・設計ドキュメント（ADR・RFC等）を探す
3. **実装の確認** — 該当する場合は WebFetch でソースコード・READMEを取得し、実際の実装・設計判断を確認する
4. **自プロジェクトとの照合**（該当する場合）— 現在のコードベースを Read/Glob/Grep で確認し、比較材料にする
5. **裏付け・突き合わせ** — 複数の情報源（公式ドキュメント、実装コード、技術記事、議論スレッド等）で内容が一致するか確認する
6. **整理と比較** — 収集した情報をテーマ・観点ごとに整理し、共通点・差異を明らかにする

## 調査の観点

- **何を解決するための設計か** — 採用された背景・課題
- **どう実装されているか** — 具体的な仕組み・データ構造・処理フロー
- **なぜその設計を選んだか** — トレードオフ・代替案との比較（ADR・議論があれば引用）
- **どこで使われているか** — 採用事例・実績
- **自プロジェクトとの関連**（該当する場合）— 現状との差分、適用する際の論点
