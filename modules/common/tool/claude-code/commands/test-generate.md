# テスト生成

**対象**: $ARGUMENTS

> - **ファイルパス指定**: そのファイルのテストを生成する
> - **関数名・箇所指定**: その関数・クラスに絞ってテストを生成する
> - **未指定**: `git diff HEAD` の変更箇所のテストを生成する

## 制約

- **対象コードの変更は行わない** — テストファイルの生成・編集のみ
- テストフレームワークはコードベースから自動判定する（package.json / go.mod / Cargo.toml 等）
- 既存テストがある場合は重複を避け、未カバーの箇所を優先する

## 実行手順

1. **対象把握** — ファイルを読み取り、関数・クラス・エクスポートを列挙する
2. **テスト環境判定** — フレームワーク・ディレクトリ規約・既存テストのスタイルを確認する
3. **テストケース設計** — 正常系・異常系・境界値・エッジケースを網羅的に洗い出す
4. **テスト生成** — 既存スタイルに揃えてテストを記述する
5. **テスト実行** — 生成したテストを実行して green であることを確認する。fail した場合は、テスト側の誤りなら修正し、対象コードのバグが疑われる場合は修正せずに報告する

## テストケース設計の観点

- **正常系** — 典型的な入力で期待通りの出力になるか
- **異常系** — 無効な入力・エラー状態で正しく失敗するか
- **境界値** — 空文字・0・null・最大値など境界での挙動
- **副作用** — DB 書き込み・外部 API 呼び出し・ファイル操作の確認
- **並行性** — race condition が起きうる箇所のテスト
