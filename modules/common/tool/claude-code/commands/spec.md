# 仕様書作成

**機能**: $ARGUMENTS

> - **機能の説明指定**: その機能の仕様書を作成する
> - **未指定**: 直前の会話コンテキストから機能を読み取って仕様化する

## 制約

- **実装・コード変更は行わない** — 仕様の文書化のみ
- 既存コードベースを参照し、命名規則・設計パターンとの整合性を確認する
- 曖昧な要件は「未決事項」として明示し、ユーザーに確認を促す
- 過度に詳細な実装指示は書かない — What を定義し、How はエンジニアに委ねる

## 実行手順

1. **要件の整理** — 引数・会話から機能の目的・スコープを把握する
2. **コードベース参照** — 関連する既存コード・型・APIを確認して設計の文脈を把握する
3. **仕様の構造化** — 機能要件・非機能要件・インターフェース・エッジケースを洗い出す
4. **未決事項の抽出** — 判断が必要な箇所を明示する

## 仕様化の観点

- **機能要件** — 何ができるようになるか（ユーザー視点）
- **非機能要件** — パフォーマンス・セキュリティ・可用性の目標値
- **インターフェース** — API・関数シグネチャ・入出力の型
- **エッジケース** — 空入力・権限エラー・競合状態など
- **スコープ外** — 今回の実装に含めないことを明示する範囲
- **テスト方針** — どのケースをテストすべきか
