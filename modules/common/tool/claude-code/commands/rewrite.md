# 完全リライト（Rust）

**対象**: $ARGUMENTS

> - **パス・箇所指定あり**: それを対象にする（説明文の場合は `Glob`/`Grep` で特定してから対象を提案冒頭で明示する）
> - **未指定**: 直前の作業対象または `git diff HEAD` を対象にする。差分が複数の無関係な変更にまたがる場合は範囲を先に確認する

## 哲学

**「現状をどう直すか」ではなく「この要件に対する最適解は何か」から考える。** 提案は常に、自分が最も良いと確信する設計に絞り込む。

- **最適解から出発する** — 既存コードの延長線上ではなく、要件だけを入力にゼロから理想の設計を組み立てる。理想から逸脱する場合は要件上の制約を明示する。「変更量が少ない」「既存に近い」は推奨理由にせず、移行コストは破壊的変更・影響範囲の欄で扱う
- **既存実装は要件の参考にとどめる** — 「何を達成したいか」だけを抽出し、実装方法・API・モジュール構成・後方互換性には縛られない。より良い設計のためなら既存の利用箇所が壊れてもよい（壊れる箇所は提案時に明示する）
- **型システムを使い切る** — 実行時チェック・命名規約・コメントで守っているものは、可能な限り型によるコンパイル時保証に置き換える
- **ただし要件は満たす** — 達成すべき機能・振る舞い（外から見た目的）は維持する。手段は全て再設計の対象
- **コードそのものが仕様書である** — 型・命名・構造で表現できることはそちらで表現し、ドキュメント・コメントは最後の手段にする

## 実行手順

このスキルは **2フェーズ** で進める。フェーズ1の提案が accept されるまで、ファイルへの書き込みは行わない。

### フェーズ1: rewrite 設計の提案（accept 待ち）

1. **要件の抽出** — 既存コードを読み、入出力・副作用・エラー条件・利用箇所を把握して「外から見て何を達成しているか」を言語化する。実装の詳細（内部構造・アルゴリズム選択）は参考にとどめ、要件と切り分ける
2. **既存設計の問題点の指摘** — なぜ改善ではなく書き直すべきか、根本的な限界を挙げる
3. **新設計の提示** — ゼロベースで考えた最良の Rust 設計を、モジュール分割・型設計・トレイト境界・主要シグネチャで示す。一意に定まる部分は確定案として書く
4. **設計判断ポイントの抽出** — 合理的な選択肢が複数あり判断が分かれる箇所を番号付きで列挙し（詳細は「出力形式」）、各論点に方針 A/B/C とメリット・デメリット・推奨案を添える。これがユーザーの意思決定の単位になる
5. **影響範囲の明示** — この書き直しで壊れる既存の利用箇所を `Grep` で確認して列挙する

ここで提案を出し、**ユーザーの判断を待つ**。設計の方向性が固まるまでは実装に着手しない。

### フェーズ2: 完全リライト（accept / 方針指定 後）

**入力形式（定型）**

| 入力 | 意味 |
|---|---|
| `accept` | 全設計判断ポイントで **推奨案** を採用する |
| `1=A,2=C` | 論点1は A、論点2は C を採用。**指定のない論点は推奨案** を採用する |
| `accept` + 補足文 | 推奨案ベースに、補足文の指示を上書きで反映する |
| 上記以外（設計そのものへの異論・追加要望・懸念） | フェーズ2には進まない。要件・新設計にフィードバックを反映し、フェーズ1の提案をやり直す |

**実行手順**

6. **採用方針の確定** — 入力を上記の形式で解釈し、各論点で採用する方針を1行で復唱する（例: `論点1=A（推奨） / 論点2=C`）
7. **書き直し** — 既存実装を捨て、確定した設計で書き直す
8. **利用箇所の追従** — 破壊的変更で壊れる呼び出し元を、新インターフェースに合わせて修正する
9. **検証** — `cargo check` → `cargo clippy` → `cargo test` の順に実行して green にし、最後に `cargo fmt` を実行する。既存テストが旧設計に依存して壊れる場合は、テストの意図（＝要件）は維持したまま新設計に沿って書き直す

## 設計の指針（Rust）

### 型・API設計

- 実行時チェック・アサーションで守る不変条件は、型（newtype・enum・ジェネリクス境界）で「不正な状態を表現不可能にする」方向に倒す。ライフサイクル・状態遷移があるなら typestate パターンも検討する
- `pub` の範囲を見直して公開 API を最小化し、不要な `clone()`/アロケーションを設計段階で排除する。イテレータ・ゼロコスト抽象を活かし、リソース管理は RAII（`Drop`）に任せて明示的な cleanup 呼び出しを利用者に要求しない

### 並行・非同期（該当する場合）

- 同期 API か非同期 API かを設計段階で決め、混在させない（ブロッキング処理は `spawn_blocking` に隔離）
- ロックより所有権・チャネルで共有を設計する。ロックを使うなら `.await` をまたがない粒度に切る

### イディオム・品質

- イテレータチェーン・`match`/`if let`/`let-else`/`matches!` の適切な使い分け、derive で済むトレイトの derive、マジックナンバー・マジック文字列の定数化

## コメント方針

コメントは書いた瞬間からコードと乖離するリスクを負う「負債」である。AI生成コードは過剰になりがちなので、書き直しの段階で徹底的に絞る。

- **WHAT/HOW は書かず、WHY だけ残す** — コードを読めばわかることは削除し、変数名・関数名で表現しきれない非自明な理由（隠れた制約・仕様上の理由・不具合への回避策など）だけを、リポジトリ内の情報だけで理解できる形で残す
- **会話・一時情報・自明なイディオムへの言及は禁止** — ユーザーとのやり取りの引き写し、Issue番号など陳腐化する外部参照、`UnsafeCell` の `Send`/`Sync` 手動実装のような標準的イディオムへの説明は書かない
- **詳細仕様は doctest に逃がす** — 引数や挙動の詳細は `# Examples` の動作可能な doctest で示し、陳腐化を防ぐ
- **`// SAFETY:`** — `unsafe` で非自明な不変条件がある場合のみ要点を簡潔に書く（`transmute` の layout 保証なら `same layout` 程度で十分）
- **doc コメント（`///`/`//!`）は公開 API のみ** — `# Errors`/`# Panics`/`# Safety` は非自明な条件がある場合のみ設け、prose の箇条書きより doctest（`should_panic`、`Err` を返す例）での実演を優先する。他の型・関数への言及は `` [`Type`] `` で intra-doc link にする

**改行（rustfmt はコメントを再整形しないため厳守）**: 1文は改行せず1行で書き、画面幅に合わせた機械的な折り返しは禁止。改行してよいのは箇条書き項目間・見出し前後・コードブロック前後、または150文字超の文を読点等の切れ目でやむを得ず区切る場合のみ。段落途中に無駄な空行を挟まない

## 出力形式

### フェーズ1: 提案

```
## 抽出した要件
[既存コードが外から見て達成していること。手段ではなく目的]

## 既存設計の問題点
[なぜ改善ではなく書き直すべきか。根本的な限界]

## 新設計
[ゼロベースで考えた最良の設計。型・トレイト境界・主要シグネチャ]
[判断が一意に定まる部分はここで確定案として記述する]

## 設計判断ポイント
判断が分かれる箇所のみを列挙する。`accept` で全て推奨案を採用、個別指定は `1=A,2=C` 形式。

### 1. [論点のタイトル]
- **A（推奨）**: [案] — 利点: [...] / 欠点: [...]
- **B**: [案] — 利点: [...] / 欠点: [...]
- **C**: [案] — 利点: [...] / 欠点: [...]
- 推奨理由: [なぜ A を推すか。1〜2行]

### 2. [論点のタイトル]
- **A（推奨）**: ...
- **B**: ...

## 破壊的変更
- [変わる API / シグネチャ / モジュール構成]

## 影響範囲（壊れる利用箇所）
- `ファイル名:行番号` — [どう壊れるか・どう追従するか]
```

**設計判断ポイントの作り方**

- **論点ごとに1番号、独立させる** — 番号間で互いに依存しないようにする
- **選択肢は A から順に、A は必ず推奨案** — 案が2つなら A/B、3つなら A/B/C
- **推奨は「最良の設計」基準で選ぶ** — 移行コスト・変更量・既存への近さは理由にせず、破壊的変更・影響範囲の欄で扱う
- **利点・欠点は具体的に** — 「柔軟」「安全」のような抽象語で終わらせず、何がどう変わるかを書く
- **真に分かれるものだけ論点にする** — 推奨が明確で他案に意味がない場合は、新設計の確定案に含める

提案後、ユーザーの判断（`accept` または方針指定）を待つ。判断が来るまで実装しない。

### フェーズ2: リライト結果（accept / 方針指定 後）

- **採用方針**: 各論点で採用した方針の復唱（例: `論点1=A（推奨） / 論点2=C`）
- **書き直したもの**: 確定した設計でのコード
- **追従修正**: 破壊的変更に合わせて修正した利用箇所
- **検証結果**: `cargo check` / `cargo clippy` / `cargo test` / `cargo fmt` の実行結果
- **要確認**: 自動検証で担保できない確認事項（実行時の挙動・外部システム連携など）。なければ「なし」
