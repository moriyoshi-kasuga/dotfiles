# 完全リライト（Rust特化）

**対象**: $ARGUMENTS

> - **ファイルパス・箇所指定**: そのファイル・箇所を対象にする
> - **未指定**: 直前の作業対象または `git diff HEAD` の変更を対象にする

## このスキルの位置づけ

| スキル | 機能変更 | 既存設計の扱い | 主な用途 |
|---|---|---|---|
| `refactor` | 行わない | 維持する | 動いているコードの内部構造を安全に整理 |
| `brushup` | 行ってよい | 意図を最優先で尊重 | 荒削りな実装を全面的に磨き上げる |
| **`rewrite`** | **大胆に行う** | **参考に留める（尊重しない）** | ゼロベースで最良の設計に完全に書き直す |

`brushup` が「動く → 良い」を実装者の意図の延長線上で目指すのに対し、
`rewrite` は **「既存実装はあくまで参考資料」** と割り切り、より良い設計を独自に組み立てて全面的に書き直す。

## 哲学

**「現状をどう直すか」ではなく「この要件に対する最適解は何か」から考える。** 提案は常に、自分が最も良いと確信する設計に絞り込む。

- **最適解から出発する** — 既存コードの延長線上で考えない。要件だけを入力に、ゼロから理想の設計を組み立てる。理想から逸脱する場合は、その理由（要件上の制約）を提案に明示する
- **妥協案・折衷案を推奨にしない** — 「無難だから」「変更量が少ないから」「既存に近いから」は推奨理由にならない。移行コストの大きさは設計を曲げる理由にせず、破壊的変更・影響範囲の欄で扱う
- **既存実装の意図は尊重しない。参考にするだけ** — 既存コードからは「何を達成したいのか（目的・要件・入出力）」だけを抽出し、「どう実装しているか」には縛られない
- **後方互換性を気にしない** — API・シグネチャ・公開インターフェース・モジュール構成を自由に変えてよい
- **破壊的変更を恐れない** — より良い設計のためなら、既存の利用箇所が壊れることを前提に進めてよい（壊れる箇所は提案時に明示する）
- **Rust の型システムを使い切る** — 実行時チェック・命名規約・コメントで守っているものは、可能な限りコンパイル時保証（型・所有権・ライフタイム）に置き換える設計にする
- **ただし要件は満たす** — 達成すべき機能・振る舞い（外から見た目的）は維持する。手段は全て再設計の対象

## 実行手順

このスキルは **2フェーズ** で進める。フェーズ1の提案が accept されるまで、ファイルへの書き込みは行わない。

### フェーズ1: rewrite 設計の提案（accept 待ち）

1. **要件の抽出** — 既存コードを読み、「外から見て何を達成しているか」を要件として言語化する
   - 入出力・副作用・エラー条件・利用箇所を把握する
   - 実装の詳細（内部構造・アルゴリズム選択）は「参考」として扱い、要件と切り分ける
2. **既存設計の問題点の指摘** — なぜ改善ではなく書き直すべきか、既存設計の根本的な限界を挙げる
3. **新設計の提示** — ゼロベースで考えた最良の設計を提案する
   - アーキテクチャ・モジュール分割・型設計・主要なインターフェースを示す
   - 設計が一意に定まる部分は確定案として書く
4. **設計判断ポイントの抽出** — 設計上、合理的な選択肢が複数あり判断が分かれる箇所を番号付きで列挙する（詳細は「出力形式」）
   - 各論点に方針 A / B / C を提示し、メリット・デメリット・推奨案を添える
   - これがユーザーの意思決定の単位になる
5. **影響範囲の明示** — この書き直しで壊れる既存の利用箇所を `Grep` で確認して列挙する

ここで提案を出し、**ユーザーの判断を待つ**。設計の方向性が固まるまでは実装に着手しない。

### フェーズ2: 完全リライト（accept / 方針指定 後）

フェーズ2は、ユーザーの入力で採用する設計方針を確定してから実行する。

**入力形式（定型）**

| 入力 | 意味 |
|---|---|
| `accept` | 全設計判断ポイントで **推奨案** を採用する |
| `1=A,2=C` | 論点1は A、論点2は C を採用。**指定のない論点は推奨案** を採用する |
| `accept` + 補足文 | 推奨案ベースに、補足文の指示を上書きで反映する |

**実行手順**

6. **採用方針の確定** — 入力を上記の形式で解釈し、各論点で採用する方針を1行で復唱する（例: `論点1=A（推奨） / 論点2=C`）
7. **書き直し** — 既存実装を捨て、確定した設計で書き直す
8. **利用箇所の追従** — 破壊的変更で壊れる呼び出し元を、新インターフェースに合わせて修正する
9. **検証** — `cargo check` → `cargo clippy` → `cargo test` の順に実行して green にし、最後に `cargo fmt` を実行する
   - 既存テストが旧設計に依存して壊れる場合は、新設計に沿って書き直す（テストの意図＝要件は維持する）

## 設計の指針（Rust）

書き直しの設計を組み立てる際の観点。

### アーキテクチャ・モジュール

- 責務ごとにモジュールを切り直す。既存のファイル分割に縛られない
- 公開 API を最小化する（`pub` の範囲を見直す）
- レイヤー間の依存方向を一方向に整える
- リソース管理は RAII（`Drop`）で設計し、明示的な cleanup 呼び出しを利用者に要求しない

### 型設計

- 不正な状態を表現できないようにする（make illegal states unrepresentable）
- parse, don't validate — 境界で一度だけ検証し、「検証済みであること」を型で表現する。生の `String` / 数値を検証せずに持ち回る設計にしない
- 有限なバリアントは `enum` で表現する（`String` / `u8` での表現をやめる）
- Newtype パターンで型安全性を高め、誤用を型で防ぐ
- 状態遷移が重要なら typestate パターン（状態ごとの型でコンパイル時に遷移を強制）を検討する
- `From` / `Into` / `TryFrom` / `FromStr` / `Display` など標準トレイトで変換・表示を標準化する
- ジェネリクス・トレイト境界で柔軟性と静的ディスパッチを両立する

### 公開 API の堅牢性

- 将来バリアント・フィールドが増えうる公開 `enum` / `struct` には `#[non_exhaustive]` を付ける
- 無視すると誤りになる戻り値・型には `#[must_use]` を付ける
- 外部実装を許さないトレイトは sealed trait パターンで閉じる
- コンストラクタで不変条件を検証し、以降は型がそれを保証する設計にする
- Rust API Guidelines に沿う — 命名規約、コレクションを持つ型の `iter` / `iter_mut` / `into_iter` 三点セット、`Debug` の全公開型への実装など
- 必須引数が多い・省略可能な設定が多い構築は builder パターンを検討する

### エラーハンドリング

- ライブラリ → `thiserror` でカスタムエラー型 / アプリ → `anyhow` で文脈付きエラー
- `?` で伝播する設計にし、`unwrap` / `expect` を本番パスから排除する
- エラーに「何が・なぜ失敗したか」のコンテキストを持たせる

### 所有権・API 境界

- 引数は用途に応じて最小限の借用で受ける（`&str` / `&[T]` / `impl AsRef<_>` / `impl Into<_>`）
- 不要な `clone()` / アロケーションを設計段階で排除する
- イテレータ・ゼロコスト抽象を活かす

### 並行・非同期（該当する場合）

- 同期 API か非同期 API かを設計段階で決め、混在させない（ブロッキング処理は `spawn_blocking` に隔離）
- ロックより所有権・チャネルで共有を設計する。ロックを使うなら `.await` をまたがない粒度に切る
- `Send` / `Sync` 境界を公開 API のシグネチャで明示し、利用側の制約を型で伝える

### イディオム・品質

- イテレータチェーン・`match` / `if let` / `let-else` / `matches!` の適切な使い分け
- derive で済むトレイトは derive する
- マジックナンバー・マジック文字列は `const` / `static` に切り出す

### コメント

コメントは「負債（コードとの乖離リスク）」である。書いた瞬間からコードと乖離するリスクを負うため、最小限にする。AI生成コードは過剰なコメントになりがちなので、書き直しの段階で徹底的に絞る。

**内容の整理**

- **不要・自明なコメントの削除**:
  - コードを読めばわかる「WHAT（何をするか）」や「HOW（どう実装するか）」はすべて削除する。
  - 変数名・関数名で表現しきれない非自明な「WHY（背景・理由）」だけを最小限残す。
- **ボイラープレート・イディオムへのコメント禁止**:
  - `UnsafeCell` を使用していることによる `Send` / `Sync` の手動実装など、標準的で自明な Rust のイディオムに対して説明コメントを書かない。
- **ドキュメントの詳細は doctest に逃がす**:
  - 詳細な仕様や引数の説明は、長文のテキストで書く代わりに `# Examples` セクションに動作可能な doctest を書いて表現する。
  - テスト可能なコード例に語らせることで、ドキュメントの陳腐化を自動的に防ぐ。
- **SAFETYコメントの最小化**:
  - `// SAFETY:` コメントは、`unsafe` ブロックを使用する際に、呼び出し側や実装側が満たすべき「非自明な」安全対策・不変条件がある場合のみ、要点に絞って簡潔に記述する。
- **doc コメント（`///` / `//!`）**:
  - 公開 API のみに限定し、目的と契約（シグネチャから読み取れない引数・戻り値の意味）に絞る。非公開 API は doc コメントではなく極力コメントなしにする。
  - エラー条件・panic 条件・`unsafe` の前提は `# Errors` / `# Panics` / `# Safety` セクションに簡潔な箇条書きで整理する。
  - 他の型・関数への言及は intra-doc link（`` [`Type`] ``）にする。

**整形 — 改行の規則（厳守）**

rustfmt はコメントを再整形しないため、不自然な改行を絶対に作らない。

- **1文＝1行を基本とする**:
  - コメント内の1文は、原則として改行せず1行で書く。
  - 画面幅に合わせるための機械的な折り返し（特に英語で `We use\n this function` や `It is safe because...\nThat` のようになるもの）は**絶対に禁止**。
- **改行してよい位置**:
  - 箇条書き（`- `）の項目間、セクション見出し（`# Examples` 等）の前後、コードブロック（` ``` `）の前後。
  - 1文が極端に長く（150文字以上）、どうしても途中で区切る必要がある場合の、読点（、）や関係代名詞の直前など、文脈が完全に切り替わる位置。
- **空白行**: 段落の途中に無駄な空行を挟まない。

## 出力形式

### フェーズ1: 提案

```
## 抽出した要件
[既存コードが外から見て達成していること。手段ではなく目的]

## 既存設計の問題点
[なぜ改善ではなく書き直すべきか。根本的な限界]

## 新設計
[ゼロベースで考えた最良の設計。アーキテクチャ・型・主要インターフェース]
[判断が一意に定まる部分はここで確定案として記述する]

## 設計判断ポイント
判断が分かれる箇所のみを列挙する。`accept` で全て推奨案を採用、個別指定は `1=A,2=C` 形式。

### 1. [論点のタイトル]
- **A（推奨）**: [案] — 利点: [...] / 欠点: [...]
- **B**: [案] — 利点: [...] / 欠点: [...]
- **C**: [案] — 利点: [...] / 欠点: [...]
- 推奨理由: [なぜ A を推すか。1〜2行]

### 2. [論点のタイトル]
- **A（推奨）**: ...
- **B**: ...

## 破壊的変更
- [変わる API / シグネチャ / モジュール構成]

## 影響範囲（壊れる利用箇所）
- `ファイル名:行番号` — [どう壊れるか・どう追従するか]
```

**設計判断ポイントの作り方**

- **論点ごとに1番号** — 1つの番号には1つの独立した判断だけを入れる。番号間は互いに依存しないようにする
- **選択肢は A から順に** — 案が2つなら A / B、3つなら A / B / C。A を必ず推奨案にする
- **推奨は「最良の設計」基準で選ぶ** — 移行コストの小ささ・変更量の少なさ・既存への近さは推奨理由にしない。それらは破壊的変更・影響範囲の欄で扱う
- **利点・欠点は具体的に** — 「柔軟」「安全」のような抽象語で終わらせず、何が・どう変わるかを書く
- **真に分かれるものだけ** — 推奨案が明確で他案に意味がないものは、論点にせず新設計の確定案に含める

提案後、ユーザーの判断（`accept` または方針指定）を待つ。判断が来るまで実装しない。

### フェーズ2: リライト結果（accept / 方針指定 後）

- **採用方針**: 各論点で採用した方針の復唱（例: `論点1=A（推奨） / 論点2=C`）
- **書き直したもの**: 確定した設計でのコード
- **追従修正**: 破壊的変更に合わせて修正した利用箇所
- **検証結果**: `cargo check` / `cargo clippy` / `cargo test` の結果
- **要確認**: 自動検証で担保できない確認事項（実行時の挙動・外部システム連携など）。なければ「なし」
