# 完全リライト（Rust特化）

**対象**: $ARGUMENTS

> - **ファイルパス・箇所指定**: そのファイル・箇所を対象にする
> - **未指定**: 直前の作業対象または `git diff HEAD` の変更を対象にする

## このスキルの位置づけ

| スキル | 機能変更 | 既存設計の扱い | 主な用途 |
|---|---|---|---|
| `refactor` | 行わない | 維持する | 動いているコードの内部構造を安全に整理 |
| `brushup` | 行ってよい | **意図を最優先で尊重** | 荒削りな実装を全面的に磨き上げる |
| **`rewrite`** | **大胆に行う** | **参考に留める（尊重しない）** | ゼロベースで最良の設計に完全に書き直す |

`brushup` が「動く → 良い」を実装者の意図の延長線上で目指すのに対し、
`rewrite` は **「既存実装はあくまで参考資料」** と割り切り、より良い設計を独自に組み立てて全面的に書き直す。

## 哲学

- **既存実装の意図は尊重しない。参考にするだけ** — 既存コードからは「何を達成したいのか（目的・要件・入出力）」だけを抽出し、「どう実装しているか」には縛られない
- **後方互換性を気にしない** — API・シグネチャ・公開インターフェース・モジュール構成を自由に変えてよい
- **破壊的変更を恐れない** — より良い設計のためなら、既存の利用箇所が壊れることを前提に進めてよい（壊れる箇所は提案時に明示する）
- **「より良くするため」だけを基準にする** — 既存の制約・慣習・経緯は、設計を良くする理由がない限り捨ててよい
- **ただし要件は満たす** — 達成すべき機能・振る舞い（外から見た目的）は維持する。手段は全て再設計の対象

## 実行手順

このスキルは **2フェーズ** で進める。フェーズ1の提案が accept されるまで、ファイルへの書き込みは行わない。

### フェーズ1: rewrite 設計の提案（accept 待ち）

1. **要件の抽出** — 既存コードを読み、「外から見て何を達成しているか」を要件として言語化する
   - 入出力・副作用・エラー条件・利用箇所を把握する
   - 実装の詳細（内部構造・アルゴリズム選択）は「参考」として扱い、要件と切り分ける
2. **既存設計の問題点の指摘** — なぜ改善ではなく書き直すべきか、既存設計の根本的な限界を挙げる
3. **新設計の提示** — ゼロベースで考えた最良の設計を提案する
   - アーキテクチャ・モジュール分割・型設計・主要なインターフェースを示す
   - 既存設計との差分、特に **破壊的変更となる点** を明示する
4. **影響範囲の明示** — この書き直しで壊れる既存の利用箇所を列挙する（`Grep` で呼び出し元を確認する）

ここで提案を出し、**ユーザーの accept を待つ**。設計の方向性が固まるまでは実装に着手しない。

### フェーズ2: 完全リライト（accept 後）

5. **書き直し** — 既存実装を捨て、フェーズ1で承認された設計で書き直す
6. **利用箇所の追従** — 破壊的変更により壊れる呼び出し元を、新インターフェースに合わせて修正する
7. **動作確認の依頼** — ビルド・テストの実行が必要な場合はユーザーに依頼する（テスト実行コマンドはプロジェクト依存のため自動実行しない）

## 設計の指針（Rust）

書き直しの設計を組み立てる際の観点。

### アーキテクチャ・モジュール

- 責務ごとにモジュールを切り直す。既存のファイル分割に縛られない
- 公開 API を最小化する（`pub` の範囲を見直す）
- レイヤー間の依存方向を一方向に整える

### 型設計

- 不正な状態を表現できないようにする（make illegal states unrepresentable）
- 有限なバリアントは `enum` で表現する（`String` / `u8` での表現をやめる）
- Newtype パターンで型安全性を高め、誤用を型で防ぐ
- `From` / `Into` / `TryFrom` / `FromStr` / `Display` など標準トレイトで変換・表示を標準化する
- ジェネリクス・トレイト境界で柔軟性と静的ディスパッチを両立する

### エラーハンドリング

- ライブラリ → `thiserror` でカスタムエラー型 / アプリ → `anyhow` で文脈付きエラー
- `?` で伝播する設計にし、`unwrap` / `expect` を本番パスから排除する
- エラーに「何が・なぜ失敗したか」のコンテキストを持たせる

### 所有権・API 境界

- 引数は用途に応じて最小限の借用で受ける（`&str` / `&[T]` / `impl AsRef<_>` / `impl Into<_>`）
- 不要な `clone()` / アロケーションを設計段階で排除する
- イテレータ・ゼロコスト抽象を活かす

### イディオム・品質

- イテレータチェーン・`match` / `if let` / `matches!` の適切な使い分け
- derive で済むトレイトは derive する
- マジックナンバー・マジック文字列は `const` / `static` に切り出す
- ドキュメントコメントは「なぜ」を書く / `unsafe` には `// SAFETY:` を付ける

## 出力形式

### フェーズ1: 提案

```
## 抽出した要件
[既存コードが外から見て達成していること。手段ではなく目的]

## 既存設計の問題点
[なぜ改善ではなく書き直すべきか。根本的な限界]

## 新設計
[ゼロベースで考えた最良の設計。アーキテクチャ・型・主要インターフェース]

## 破壊的変更
- [変わる API / シグネチャ / モジュール構成]

## 影響範囲（壊れる利用箇所）
- `ファイル名:行番号` — [どう壊れるか・どう追従するか]
```

提案後、ユーザーの accept を待つ。accept されなければ実装しない。

### フェーズ2: リライト結果（accept 後）

- **書き直したもの**: 新しい設計でのコード
- **追従修正**: 破壊的変更に合わせて修正した利用箇所
- **要確認**: ビルド・テストでの動作確認を依頼する項目
